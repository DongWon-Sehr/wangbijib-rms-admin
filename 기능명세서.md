# 왕비집 예약관리 시스템 v2.0.60 기능명세서

## 1. 개요 (Overview)

* **프로젝트명**: 왕비집 예약관리 시스템 고도화
* **목적**: 왕비집 전 지점의 예약 관리, 슬롯 최적화, 예약금 관리 및 고객 응대 자동화 효율성 증대.
* **플랫폼**: Google Workspace 기반 웹 애플리케이션 (Google Sheets + Google Apps Script).
* **배포 설정**:
  * 실행 권한: `Me` (소유자 권한으로 스크립트 실행)
  * 접근 권한: `Every Google Account` (모든 구글 계정 사용자 접근 가능 - 내부 인증 로직으로 통제)
* **핵심 가치**: "예약 슬롯의 유기적 확보" (예약 확정 시 위젯 캘린더 슬롯을 자동으로 확보하여 매출 증대 기여).

## 2. 시스템 아키텍처 & 기술 스택

### 2.1. Frontend (SPA)
* **Framework**: `Vue.js 3` (Composition API, CDN 방식).
* **CSS Framework**: `Tailwind CSS` (CDN 방식).
* **Icons**: `Phosphor Icons`.
* **Rich Text Editor**: `Quill.js` (메일 템플릿 편집용).
* **Notification**: KaKao JS SDK (알림톡 전송).
* **구조 전략**:
  * **Template 분리**: `index.html` 유지보수를 위해 `LayoutHeader.html`, `ViewList.html`, `ModalReservation.html` 등 컴포넌트 단위로 파일 분리 후 `include()` 처리.
  * **데이터 캐싱 & 동기화**:
    * **Initial Load**: 최초 진입 시 스피너를 띄우고 전체 예약 데이터 및 설정(Branch Map, Slot Map, Mail Templates)을 메모리에 로드.
    * **Reactive Update**: 데이터 수정 시 로컬 상태(State)를 즉시 업데이트하여 UI 반응성 확보 후 백그라운드 서버 동기화.
    * **Background Sync**: 1분 주기로 서버 데이터를 조용히 페치(Fetch)하여 변경 사항을 로컬 상태에 병합. (예약 상세 모달이 열려 있는 경우에도 작동하여, 누락되었던 `email_thread_id`가 보정되면 메일 발송 버튼을 자동 활성화하며, 헤더 우측에 1초 단위의 동기화 카운트다운 UI를 제공하여 가시성을 높임)
  * **모바일 최적화**: 터치 기반의 **당겨서 새로고침(Pull-to-Refresh)** 기능 구현.

### 2.2. Backend (Google Apps Script)
* **패턴**: `Singleton Pattern` 적용. `Config.gs`, `Util.gs`를 포함한 모든 서비스 파일은 `const` Service = {} 객체 리터럴 구조로 변경하여 인스턴스 관리 비용 제거. (단, `ReservationBatch.gs`, `WebApi.gs`는 함수형 유지)
* **API Wrapper 패턴**: `WebApi.gs` 내 `_executeApi(apiName, action, params)` 래퍼 함수 도입.
  * 모든 API 요청의 진입/성공/실패/에러를 통일된 포맷으로 로깅.
  * **Error Handling**: Service 계층에서 에러 발생 시 래퍼가 포착하여 표준 실패 응답 반환.
* **데이터 액세스**: **Header Name 기반 Object Mapping** (ORM 유사 방식) 적용. `Util.getSheetDataAsObjects(sheetName)`.
* **인증(Auth)**: **Google Session 인증** (`Session.getActiveUser().getEmail()` 사용). 기존 비밀번호 방식 폐기.
* **템플릿 엔진**: **구글 시트(`mail_template`)**를 DB로 사용하여 HTML 템플릿을 관리. WebApi를 통해 조회 및 업데이트.

### 2.3. 디자인 시스템 (Color Palette)
* **Main Color (Royal Red)**: `#8B0000` - 왕비집의 정체성.
* **Sub Color (Cool Gray)**: `#1F2937` - 텍스트 및 네비게이션.
* **Accent Color (Amber)**: `#D97706` - 중요 알림, 대기 상태 뱃지.
* **Background**: `#F3F4F6` - 대시보드 배경.

## 3. 통신 프로토콜 (API Interface)

### 3.1. 표준 응답 포맷 (Standard Response)
`Util.createResponse()`를 통해 아래 JSON 구조 준수.
```json
{
  "success": boolean,
  "data": any,
  "message": string
}
```

### 3.2. Util 메서드 정의
* **`Util.createResponse(success, data, message)`**: 표준 응답 객체 생성.
* **`Util.getSheetDataAsObjects(sheetName)`**: 시트 데이터를 JSON 객체 배열로 반환.

## 5. 상세 기능 명세

### 5.1. 공통 기능
* **Custom Alert/Confirm**: 시스템 전반에 통일된 디자인의 커스텀 모달 사용.
* **Global Spinner**: 초기 로딩 및 주요 처리 시 시각적 피드백 제공.

### 5.2. 예약 리스트 (Tab: List)
* **당겨서 새로고침 (Pull-to-Refresh)**: 모바일 리스트 최상단에서 아래로 당겨 최신 데이터 갱신 지원.
* **검색 & 필터**:
  * 기간, 지점, 상태(대기/예약/취소), 예약금 상태 복합 필터링.
  * 상태별 행 배경색 차별화 (취소: 회색, **대기: 옅은 노란색**).
* **인라인 상태 변경**: 리스트에서 예약 상태 및 예약금 상태를 즉시 변경 가능.

### 5.3. 예약 상세/수정 (Modal)
* **데이터 정합성 보호**: 수정 사항이 있는 상태에서 저장 없이 알림톡 전송 시 **"변경사항 저장 필요"** 안내문 노출.
* **읽음 처리**: 모달 오픈 시 또는 알림톡 전송 시 자동 읽음(`is_read=true`) 처리.
* **상태 관리**: 모달 내부에서 지점, 예약 상태(대기/예약/취소)를 직접 수정 및 저장 가능.
* **UI 최적화**: 
  * 수정 가능 필드는 흰색 배경, 읽기 전용 필드는 회색 배경으로 명확히 구분.
  * 메일 발송, 알림톡 전송 등 모달 내부 작업 시 화면 전체 스피너 대신 모달 내부 스피너만 작동하도록 하여 사용자 경험(UX) 개선.

### 5.4. 시스템 설정 (Tab: Settings)
* **탭 네비게이션 최적화**: 사용 빈도를 고려하여 탭 순서를 재배치(슬롯 관리 -> 메일 템플릿 -> 지점 관리 -> 사용자 관리)하고, 최초 진입 시 '슬롯 관리'가 기본으로 표시됨.
* **커스텀 슬롯 일괄 관리**:
  * 특정 날짜/지점의 모든 시간대를 한 번에 수정하는 일괄 입력 기능.
  * **슬롯 수와 활성화 여부(ON/OFF)를 동시 적용** 가능하도록 고도화.
  * **안전한 데이터 처리**: 동시성 문제를 방지하기 위해 일괄 저장(Batch API) 방식 적용.
* **단일 슬롯 수정**: 개별 시간대 클릭 시 기존 설정값을 자동으로 불러오며(Prefill), 새로 설정 시 비활성화(OFF)를 기본값으로 제공.

### 5.5. 메일 템플릿 관리 (Tab: Settings > Mail)
* **시각적 편집 (WYSIWYG)**: Summernote 에디터를 탑재하여 HTML 본문을 직접 편집 가능.
* **치환자 시스템 (Placeholder)**:
  * 예약 데이터(고객명, 방문일, 예약금, 예약금 결제 링크 등)를 동적으로 삽입하기 위해 `[[keyword]]` 형식의 치환자 사용.
  * **에디터 뱃지화**: 편집 편의성을 위해 에디터 내에서는 치환자를 뱃지(Badge) UI로 표시하며, 저장 시 자동으로 텍스트 포맷으로 변환하여 DB에 저장.
* **미리보기**: 편집 중인 템플릿이 실제 데이터와 결합되었을 때의 결과물을 실시간으로 확인하는 미리보기 기능 제공.
* **삭제 정책**: 템플릿 삭제 시 복구가 불가능한 실제 행 삭제(Hard Delete)를 수행함.

## 6. 비즈니스 로직 (Business Logic)

### 6.1. 예약 상태 라이프사이클
* **신규 접수**: 모든 외부 신규 예약은 초기 상태가 **`pending` (대기)**으로 설정됨.
* **자동 예약 확정**: 관리자가 상세 화면이나 리스트에서 **알림톡(카카오톡) 발송 완료** 버튼을 누를 경우, 시스템이 이를 인지하여 상태를 **`confirm` (예약 확정)**로 자동 전환.
* **수동 상태 변경**: 관리자의 필요에 따라 언제든 대기/예약/취소 상태로 전환 가능.

### 6.2. 지메일 라벨 동기화
* 예약 상태 변화에 따라 연동된 지메일 스레드의 라벨이 실시간으로 동기화됨.
  * 대기 (`pending`): `매장/예약/대기`
  * 예약 (`confirm`): `매장/예약/완료`
  * 취소 (`cancel`): `매장/예약/취소`

### 6.3. 예약금 정책
* **대상**: 9인 이상 예약 시 자동 감지.
* **자동화**: `ReservationBatch` 실행 시 9인 이상이면 `deposit_status='pending'` 설정 및 안내 로직 작동.
* **계산금액**:
  * 9인 이상 ~ 19인 이하: 10만원
  * 20인 이상 ~ 29인 이하: 20만원
  * 30인 이상 ~ 39인 이하: 30만원
  * 40인 이상 ~ 49인 이하: 40만원
  * 50인 이상 ~ 59인 이하: 50만원
  * 60인 이상 : 에러 (메일 자동발송 안함)

### 6.4. 슬롯 확보 및 동기화
* 예약 상태가 `cancel` (취소)가 되면 점유하고 있던 캘린더 이벤트를 즉시 삭제하고 해당 슬롯을 다시 확보함.
* **투트랙(Two-Track) 캘린더 자동 동기화**: 기본 슬롯(Default) 설정 변경 시 앱스 스크립트 실행 시간 제한(6분)을 우회하고 미래 예약(향후 40일)을 안전하게 제어하기 위한 배치 처리 적용. (커스텀 슬롯 설정이 우선 적용됨)
  * **Track 1 (실시간 백그라운드 큐)**: 설정 저장 시 3초 뒤 백그라운드 1회성 트리거(`triggerBackgroundSlotSync`)를 구동시켜 향후 40일 치 캘린더 이벤트를 즉시 동기화.
  * **Track 2 (새벽 롤링 배치)**: 매일 새벽 시간 기반 트리거(`processDailySlotSync`)가 모든 활성 지점의 향후 40일 치 슬롯을 훑으며 누락된 미래 일자를 캘린더에 동기화.
* **Google Calendar Batch API 최적화**: 다량의 캘린더 이벤트 생성/삭제 시 발생하는 API 호출 제한 및 스크립트 실행 지연을 방지하기 위해 단일 API 호출 대신 **Batch Request** 방식으로 묶어서 전송.
  * `multipart/mixed` 페이로드로 여러 요청을 묶음.
  * 한 번에 최대 **50개 단위(Chunk)**로 분할하여 전송하며, 청크 간 지연(500ms)을 주어 Rate Limit을 우회.
  * 백그라운드 큐 처리 시 한 번에 동기화하는 기간을 기존 20일에서 **10일(Window)**로 축소하여 메모리 및 실행 안정성을 극대화.
* **백그라운드 배치 안정성 강화**:
  * **동시성 제어 (LockService)**: 큐(`PropertiesService`) 접근 시 스크립트 락을 적용하여 여러 프로세스가 동시에 데이터를 수정할 때 발생하는 데이터 경합 및 유실을 원천 차단함.
  * **트리거 강제 갱신 정책**: 큐 등록 시 기존에 등록된 트리거(비활성 상태 포함)를 선제적으로 삭제하고 항상 새로운 트리거를 생성하여 확실한 실행을 보장함.
  * **에러 내성 (Resilience)**: 동기화 과정에서 일시적인 API 오류가 발생하더라도 대기 중인 전체 큐를 삭제하지 않고 보존하여, 다음 배치나 트리거 실행 시 재시도할 수 있도록 개선함.
  * **트리거 생성 부하 최적화 (Batch Enqueue)**: 여러 지점의 동기화가 동시에 필요할 때, 지점별로 트리거를 매번 생성/삭제하지 않고 모든 대상을 큐에 먼저 적재한 후 트리거는 단 1회만 갱신하도록 개선하여 구글 API 호출 제한 및 스크립트 중단(Canceled) 문제를 해결함.

### 6.5. Gmail 스레드 ID 보정 로직 (Auto-Repair)
* **대상**: `email_thread_id`가 누락된 최근 **24시간 이내**의 활성 예약 건.
* **배치 실행**: `ReservationBatch` 실행 시마다 위 대상을 필터링하여 최대 **10건**씩 재검색 및 보정을 시도함.
* **목적**: Gmail의 검색 인덱싱 지연으로 인해 최초 등록 시 ID를 확보하지 못한 경우를 사후 복구하여 시스템 무결성을 보장함.
